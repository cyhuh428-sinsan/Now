# NowNote 2.3 통합 작업지시서

## 2026-06-12 현재 진행 기준

이 문서는 NowNote 2.3의 실행 기준 문서다.

현재 상태는 **2.3 완료가 아니라 2.3 본 작업 진입 단계**다. 인증/서버 연결 정리는 1차 반영되었지만, 운영 서버 실사용 검증, 앱 잔여 변경 정리, 설치형 저장 검증 재확인, 메신저 2.3 확장 구현이 남아 있다.

### 현재 반영된 항목

- 서버/API 인증 역할 정리 1차 반영
- Web ID/password 로그인 기준 유지
- 설치형 프로그램 API 연결 기준 유지
- 앱/설치형 접속 토큰 명칭 1차 정리
- 구형 개인 서버 API 토큰의 고급 설정 분리 1차 반영
- Web/설치형 도움말 토큰 표현 정리
- 서버 preflight 문서 검증 실패 2건 수정
- 관련 변경 사항 GitHub `main` push 완료

### 아직 완료로 볼 수 없는 항목

- 운영 서버 `https://nownote.sinsan.kr` 기준 Web 실제 동작 검증 미완료
- 설치형 프로그램 실제 연결 테스트 미완료
- 앱 실제 에뮬레이터/운영 서버 연결 검증 미완료
- 앱 쪽 잔여 변경 정리 및 커밋 필요
- 설치형 `npm run check:storage` 재검증 필요
- 메신저 2.3 확장 기능 미구현
- exe/apk 2.3 릴리즈 asset 업로드 미완료

### 현재 우선순위

1. 앱 잔여 변경 검토, 테스트, 커밋, push
2. 서버에서 `git pull` 후 `deploy_local.sh --base-url https://nownote.sinsan.kr` 재실행
3. 운영 서버 Web 로그인, 계정, 토큰 발급, 공유 문서 조회 확인
4. 설치형 프로그램 기존 실행 프로세스 종료 후 `npm run check:storage` 재검증
5. 설치형에서 `https://nownote.sinsan.kr` + 사용자 ID + 앱/설치형 접속 토큰으로 연결 테스트
6. 앱 에뮬레이터에서 `https://nownote.sinsan.kr` + 사용자 ID + 앱/설치형 접속 토큰으로 연결 테스트
7. 메신저 2.3 확장 설계 확정 후 구현
8. exe/apk 빌드 및 GitHub Release asset 업로드

### 작업 분리 기준

서버/API, Web/설치형, 앱은 같은 저장소에 있지만 서로 다른 실행 산출물이다. 한 영역 수정 후에는 해당 영역 검증을 먼저 통과시킨 뒤 push한다. 다른 영역의 미커밋 변경이 남아 있으면 섞어서 커밋하지 않는다.

### Git 기준

- 수정 이후에는 반드시 커밋하고 `origin/main`에 push한다.
- exe/apk를 빌드한 경우 GitHub Release asset으로 별도 업로드한다.
- 운영 서버 테스트는 서버에서 `git pull origin main` 후 수행한다.

## 2.3 핵심 목표

2.3은 메신저 기능만 추가하는 버전이 아니다.

2.3의 핵심 목표는 아래 순서다.

1. Web과 설치형 프로그램을 메모앱답게 가볍고 안정적으로 만든다.
2. 제목, 본문, 검색, 본문찾기, 단축키, Tab 들여쓰기를 안정화한다.
3. 작은 창에서도 편집 가능한 화면 구조로 정리한다.
4. 인증/서버 연결 구조를 사용자 화면에서 단순하게 정리한다.
5. 앱 서버 설정과 조회 문제를 설치형 기준에 맞춰 정리한다.
6. 그 다음 메신저 첨부, 사진, 일부 그룹원 채팅방 기능을 추가한다.
7. 서버, Web, 설치형, 앱을 분리 검증한 뒤 같은 버전으로 릴리즈한다.

## 작업 원칙

- 메모 입력 안정화가 모든 기능보다 우선이다.
- Web 실행만으로 시스템 전체가 느려지면 2.3 완료로 보지 않는다.
- 설치형 프로그램은 별도 exe로 유지한다.
- 설치형 프로그램은 WebView 눈속임으로 대체하지 않는다.
- 설치형은 Web 전용 기능을 제외하고 Web과 같은 디자인/기능 흐름을 유지한다.
- 앱은 Web/설치형 수정이 자동 반영되지 않는 별도 Flutter 앱으로 본다.
- Web 세션 전용 기능을 설치형/앱에 그대로 넣지 않는다.
- 운영자가 CLI나 DB를 직접 만져야만 확인 가능한 기능은 완료로 보지 않는다.

## 개발 순서

### 0단계: 기준 확정

목표:

- 2.3 통합 기준을 확정한다.
- 각 작업자가 같은 기준으로 작업하게 한다.

작업:

- 이 문서를 기준 문서로 사용한다.
- 인증/서버 연결 기준은 `NOW_2_3_AUTH_CONNECTION_WORK_ORDER.md`를 따른다.
- 릴리즈 기준은 `NOW_2_3_RELEASE_GOVERNANCE_CHECKLIST.md`를 따른다.

완료 조건:

- 서버/API, Web/설치형, 앱 작업 범위가 분리되어 있다.
- 2.3에서 먼저 할 작업과 나중에 할 작업이 구분되어 있다.

### 1단계: Web/설치형 메모 코어 안정화

목표:

- 제목, 본문, 검색, 본문찾기, 단축키가 안정적으로 동작하게 한다.

작업:

- 제목 입력 잠김 문제 확인 및 수정
- 본문 입력 포커스 안정화
- 전체 검색 입력 안정화
- 본문찾기 입력 안정화
- `Ctrl+F`는 전체 검색으로 고정
- `Ctrl+Shift+F`는 현재 메모 본문찾기로 고정
- Tab 들여쓰기 동작 확인
- Tab 들여쓰기 설정값 적용 확인
- IME 입력 중 렌더링/포커스 강탈 방지

완료 조건:

- Web에서 제목 입력 가능
- Web에서 본문 입력 가능
- Web에서 검색 입력 가능
- Web에서 본문찾기 입력 가능
- 설치형에서 제목 입력 가능
- 설치형에서 본문 입력 가능
- 설치형에서 검색 입력 가능
- 설치형에서 본문찾기 입력 가능
- Web/설치형 모두 `Ctrl+F`, `Ctrl+Shift+F` 기준 통과
- 설치형에서 Tab 들여쓰기 동작

### 2단계: Web/설치형 경량 런타임 정리

목표:

- Web 실행 시 시스템 전체가 느려지는 문제를 줄인다.
- 메모 입력과 부가 기능을 분리한다.

작업:

- 전체 render 호출 지점 분류
- 입력 중 전체 render 금지
- 메신저 polling과 메모 저장 루프 분리
- unread count를 가벼운 갱신으로 분리
- 메신저가 닫혀 있으면 메시지 목록 조회 중지
- 서버 동기화, 공유 조회, 메신저 갱신을 scheduler 기준으로 정리
- 비활성 탭 backoff 적용
- 같은 응답이면 DOM 갱신 생략

완료 조건:

- Web idle 상태에서 반복 렌더링이 없어야 한다.
- 메신저 닫힘 상태에서 메시지 목록 polling이 없어야 한다.
- 제목/본문 입력 중 서버 갱신으로 포커스가 이동하지 않아야 한다.
- 검색/본문찾기 입력 중 한 글자만 입력되는 문제가 없어야 한다.

### 3단계: 작은 창과 UI 정리

목표:

- 메모 프로그램을 작게 띄워도 편집이 깨지지 않게 한다.

작업:

- 상단 긴 텍스트 버튼을 아이콘 또는 더보기 메뉴로 정리
- 탭 조작 버튼을 아이콘 중심으로 정리
- 설정에서 자주 쓰는 아이콘 표시 여부를 선택 가능하게 유지 또는 추가
- 작은 창에서 사이드바/트리/본문 우선순위 정리
- 본문 편집 영역 최소 확보
- 폰트 기준 적용
  - 최상위 제목: 14
  - 제목: 12
  - 본문: 11 또는 10
  - 설명: 9

완료 조건:

- 작은 창에서도 제목과 본문 편집 가능
- 버튼이 본문 영역을 과도하게 차지하지 않음
- Web과 설치형의 기본 디자인 흐름이 동일
- Web 전용 기능은 설치형에서 숨김
- 폰트 기준이 주요 화면에 적용됨

### 4단계: 인증/서버 연결 정리

목표:

- 현재 많은 인증 구조를 사용자 화면에서 단순하게 정리한다.

확정 기준:

- Web은 사용자 ID/비밀번호 로그인
- 설치형은 서버 API 연결
- 앱은 서버 API 연결
- 설치형/앱 기본 연결 수단은 서버 주소, 사용자 ID, 앱/설치형 접속 토큰
- 구형 개인 서버 API 토큰은 기본 흐름에서 제거하고 고급 설정으로 숨김

서버 및 API 작업:

- `/api/v1/auth/web-login` 유지
- `/api/v1/auth/web-session` 유지
- `/api/v1/auth/token-login` 유지
- 앱/설치형 데이터 API에서 `X-Now-User-Token` 검증 유지
- `NOW_API_TOKEN`은 구형 개인 서버 호환용으로 유지
- 조회 실패 시 원인 구분이 가능하게 응답/로그 확인

Web 및 설치형 작업:

- Web 로그인 화면에 토큰 입력 노출 금지
- Web에서 앱/설치형 접속 토큰 발급 안내 명확화
- 설치형에서 구형 개인 서버 API 토큰을 고급 설정으로 이동
- 설치형에서 앱/설치형 접속 토큰 설명 정리

앱 작업:

- `사용자별 접속 토큰`을 `앱/설치형 접속 토큰`으로 변경
- `API 토큰`을 `구형 개인 서버 API 토큰`으로 변경
- 구형 개인 서버 API 토큰은 고급 설정으로 숨김
- 기기 ID도 가능하면 고급 설정으로 이동
- 설치형과 설명 기준을 맞춤

완료 조건:

- Web은 ID/password 로그인 유지
- 설치형은 API 연결 유지
- 앱은 API 연결 유지
- 공용 서버 일반 사용자가 API 토큰을 입력하지 않아도 됨
- 문서와 화면 설명이 충돌하지 않음

### 5단계: 앱 조회 문제 확인과 정리

목표:

- 앱에서 조회가 잘 안 되는 원인을 확인하고, 앱 서버 설정 흐름을 2.3 기준에 맞춘다.

작업:

- 서버 주소 접근 가능 여부 확인
- 사용자 ID와 서버 `owner_id` 일치 여부 확인
- 앱/설치형 접속 토큰 유효성 확인
- `/api/v1/auth/token-login` 연결 테스트 확인
- 동기화 요청에 `X-Now-User-Token` 포함 여부 확인
- 서버에서 내려온 메모가 앱 로컬 DB에 반영되는지 확인
- 앱이 Web 세션 전용 공유 문서나 그룹 메모를 기대하고 있지 않은지 확인
- `now_app/scripts/verify_mobile_surface.py` 검증 보강

완료 조건:

- 앱 조회 실패 원인이 확인됨
- 앱 서버 설정 문구가 설치형 기준과 맞음
- 모바일 정적 검증 통과
- Flutter 테스트 통과

### 6단계: 메신저 2.3 확장

목표:

- 기존 2.2 전체 그룹 메신저를 유지하면서 파일/사진 첨부와 일부 그룹원 채팅방을 추가한다.

서버 및 API 작업:

- 기존 `group_messages` 보존
- 새 메신저 room 구조 도입 여부 확정
- 전체 그룹 기본 채팅방 유지
- 일부 그룹원 채팅방 API 추가
- 채팅방 참여자 권한 검증
- 메시지 조회/전송 API
- unread API
- 첨부 업로드 API
- 첨부 다운로드 API
- 첨부 정책 API
- 파일 크기 제한
- 확장자/MIME 제한
- 서버 저장소 경로 안전화
- 운영 점검 API 추가

## 최종 완료 기준

2.3은 아래 조건을 모두 만족해야 완료다.

- Web이 가볍게 동작한다.
- 설치형 프로그램이 작은 창에서도 깨지지 않는다.
- 제목, 본문, 검색, 본문찾기 입력이 안정적이다.
- `Ctrl+F`, `Ctrl+Shift+F`, Tab 들여쓰기가 기준대로 동작한다.
- Web은 ID/password 로그인 기준을 유지한다.
- 설치형/앱은 앱/설치형 접속 토큰 기반 API 연결 기준을 유지한다.
- 구형 개인 서버 API 토큰은 기본 화면에서 숨겨져 있다.
- 앱 조회 문제 원인이 확인되고 필요한 수정이 반영되어 있다.
- 전체 그룹 메신저가 유지된다.
- 일부 그룹원 채팅방과 파일/사진 첨부가 동작한다.
- 첨부 보안 정책이 적용된다.
- 운영 화면/API로 저장소와 이상 상태를 확인할 수 있다.
- 서버/Web/설치형/앱 검증이 통과한다.
- exe와 apk가 GitHub Release asset으로 업로드되어 있다.
