# NowNote 2.3 전체 구조 재설계서

## 문서 상태

이 문서는 NowNote 2.3 개발의 상위 기준서다.

2.3은 단순히 메신저 기능을 확장하는 버전이 아니라, Web과 설치형 프로그램을 메모앱답게 가볍고 안정적으로 다시 정리한 뒤 메신저 첨부와 채팅방 기능을 올리는 버전으로 정의한다.

기존 문서와의 관계는 아래와 같다.

- `NOW_2_3_LIGHTWEIGHT_RUNTIME_REDESIGN.md`: 경량 런타임 기준 문서
- `NOW_2_3_GROUP_MESSENGER_ATTACHMENTS_AND_ROOMS_DESIGN.md`: 메신저 첨부와 채팅방 후보 설계 문서
- 이 문서: 2.3 전체 구조, 개발 순서, 검증 기준을 확정하는 상위 설계서

## 확인된 문제

아래 문제는 사용자 확인이 끝난 현재 기준의 사실로 본다.

- Web 실행 시 시스템 전체가 느려지는 경우가 있다.
- Web과 설치형 프로그램 모두 입력창 포커스가 안정적이지 않다.
- 검색 팝업 입력이 되지 않거나 한 글자만 입력되는 문제가 있다.
- 본문찾기 입력이 되지 않거나 한 글자만 입력되는 문제가 있다.
- 제목 수정이 잠긴 것처럼 동작하는 경우가 있다.
- 단축키가 기대대로 동작하지 않는다. NowNote 기준으로 `Ctrl+F`는 통합 검색, `Ctrl+Shift+F`는 현재 메모 본문찾기에 연결되어야 한다.
- 작은 창에서 상단 버튼과 편집 영역이 깨진다.
- 설치형 프로그램은 별도 exe로 유지해야 하며, Web을 눈속임으로 감싼 형태가 아니어야 한다.
- 설치형 프로그램은 Web 전용 기능을 제외하고 디자인과 기능 흐름이 Web과 같아야 한다.
- 메신저, 동기화, 공유 조회 같은 부가 기능이 메모 입력 경험을 방해하면 안 된다.

## 2.3 목표

2.3의 목표는 세 가지다.

1. 메모 코어 안정화
   - 제목 입력, 본문 입력, 검색, 본문찾기, 탭, 로컬 저장을 최우선으로 안정화한다.

2. 경량 런타임 구조 전환
   - 렌더링, 서버 동기화, 메신저 갱신, 공유 조회를 분리해 Web 실행만으로 시스템이 느려지지 않게 한다.

3. 메신저 확장 기반 마련
   - 전체 그룹 메신저를 유지하면서, 2.3에서 채팅방, 일부 그룹원 대화, 파일/사진 첨부를 안전하게 추가할 수 있는 구조를 만든다.

## 제외 범위

2.3에서 아래 항목은 하지 않는다.

- 설치형 프로그램을 WebView 포장만으로 대체
- 기존 데이터 전체 재작성
- 기존 2.2 그룹 메시지 삭제
- 모바일 앱 메신저 확장
- 음성/영상 통화
- 대용량 파일 전송 서비스 수준의 기능
- 운영자가 CLI나 DB를 직접 만져야 하는 흐름

## 구조 원칙

### 1. 메모 코어 우선

메모 코어는 어떤 부가 기능보다 우선한다.

메모 코어에 포함되는 기능은 아래로 제한한다.

- 트리 선택
- 제목 수정
- 본문 수정
- 열린 탭
- 로컬 저장
- 검색 팝업 입력
- 본문찾기 입력
- 기본 단축키

메모 코어는 메신저, 서버 동기화, 그룹 공유 목록, unread count 때문에 전체 렌더링되면 안 된다.

### 2. 입력과 포커스는 독립 관리

입력창은 렌더링 함수가 직접 포커스를 빼앗지 못하게 한다.

2.3에서 모든 입력은 아래 기준을 따른다.

- `input` 이벤트 안에서 다른 입력창이나 본문 편집기로 강제 `focus()`하지 않는다.
- 검색 결과 갱신은 입력값을 보존해야 한다.
- 본문찾기 입력 중에는 본문 선택 표시를 위해 편집기로 포커스를 이동하지 않는다.
- 이전/다음 결과 버튼이나 Enter 탐색 때만 본문 선택을 수행한다.
- `Ctrl+F`는 통합 검색을 연다.
- `Ctrl+Shift+F`는 현재 메모 본문찾기를 연다.

### 3. 렌더링은 필요한 영역만 갱신

전체 `render()`를 모든 상태 변경의 기본 처리로 쓰지 않는다.

렌더링 영역을 아래처럼 분리한다.

| 영역 | 렌더링 트리거 |
| --- | --- |
| 사이드바 | 메모 목록, 설정 변경 |
| 트리 패널 | 트리 구조 변경 |
| 편집기 제목 | 선택 메모 변경 또는 제목 입력 |
| 편집기 본문 | 선택 메모 변경 |
| 탭바 | 열린 탭 상태 변경 |
| 검색 팝업 | 검색어, 범위, 정렬 변경 |
| 본문찾기 | 검색어와 현재 메모 변경 |
| 메신저 | 메신저 패널 표시 또는 채팅방 변경 |
| 설정 | 설정 팝업 표시 또는 설정 변경 |

### 4. 백그라운드 작업은 하나의 스케줄러로 통제

기능별 독립 polling을 금지한다.

2.3에서는 하나의 `runtime scheduler`가 아래 작업을 통제한다.

- 서버 동기화
- unread count 조회
- 열린 메신저 방 메시지 조회
- 그룹 목록 조회
- 공유 문서 갱신

스케줄러 기준은 아래와 같다.

- 사용자가 입력 중이면 무거운 작업을 지연한다.
- 브라우저 탭이 비활성 상태면 갱신 주기를 늦춘다.
- 메신저가 닫혀 있으면 메시지 목록은 조회하지 않는다.
- 같은 응답이면 DOM을 갱신하지 않는다.
- 실패 재시도는 backoff를 적용한다.
- 동시에 여러 요청을 보내지 않는다.

### 5. Web과 설치형은 같은 UX, 다른 런타임

설치형 프로그램은 별도 Electron exe로 유지한다.

공통으로 유지할 것:

- 전체 화면 구조
- 사이드바 구조
- 트리/탭/편집기 구조
- 설정 화면 구조
- 검색, 본문찾기, 단축키 흐름
- 작은 창 대응 방식

분리할 것:

| 구분 | Web | 설치형 |
| --- | --- | --- |
| 원본 데이터 | 서버 공유 문서 | PC 로컬 저장소 |
| 로그인 | 사용자 ID/비밀번호 | 기본 로컬 실행, 서버 연결은 선택 |
| 그룹 참가 | 포함 | 제외 |
| 그룹 메신저 | 포함 | 제외 |
| 앱/설치형 토큰 발급 | 포함 | 제외 |
| 서버 동기화 | 서버 공유 문서 중심 | 사용자가 공유 선택한 메모 중심 |

## 2.3 화면 구조

### 기본 화면

기본 화면은 아래 구조로 고정한다.

```text
App Shell
├─ Rail: 아이콘 중심 빠른 이동
├─ Sidebar: 검색, 주요 보기, 관리 메뉴
└─ Workspace
   ├─ Header: 현재 위치와 최소 조작
   ├─ Tree Panel: 주제/분류/메모 목록
   └─ Editor Panel
      ├─ Tabs: 열린 메모
      ├─ Title Row: 제목 입력 + 저장 상태 + 더보기
      ├─ Find Row: 본문찾기
      ├─ Metadata Row: 통계/태그/백링크 요약
      └─ Body: 본문 편집
```

### 작은 창 기준

작은 창에서는 본문과 제목을 우선한다.

- 긴 텍스트 버튼은 편집기 상단에 직접 배치하지 않는다.
- 탭 조작은 아이콘 버튼으로 표시한다.
- 메모 조작은 기본적으로 더보기 메뉴에 넣는다.
- 사용자가 설정하면 자주 쓰는 조작만 아이콘으로 밖에 표시한다.
- 일정 폭 이하에서는 사이드바와 트리 패널을 접을 수 있어야 한다.
- 본문 편집 영역은 항상 남아야 한다.

## 입력/단축키 설계

### 입력 관리자

2.3에서는 입력 포커스를 별도 규칙으로 관리한다.

입력 관리자 책임:

- 현재 활성 입력 영역 추적
- 입력 중 렌더링 지연
- 검색/본문찾기 포커스 보존
- IME 입력 중 조합 상태 보호
- 팝업 열기 후 최초 포커스 1회만 수행

금지:

- `input` 이벤트 안에서 반복 `focus()`
- 렌더링 후 무조건 편집기 포커스
- 검색 결과 선택 표시를 위해 입력창 포커스 강탈

### 단축키 관리자

단축키는 전역 키다운에서 직접 처리하지 않고, 현재 컨텍스트 기준으로 해석한다.

우선순위:

1. 단축키 설정 캡처 중
2. 검색/본문찾기/팝업 입력 중
3. 본문 편집 중
4. 일반 화면

기본 단축키:

| 단축키 | 동작 |
| --- | --- |
| `Ctrl+F` | 통합 검색 |
| `Ctrl+Shift+F` | 현재 메모 본문찾기 |
| `Ctrl+K` | 빠른 전환 |
| `Ctrl+N` | 새 주제 |
| `Ctrl+Shift+N` | 아래에 추가 |
| `Esc` | 현재 팝업 또는 본문찾기 닫기 |

messengerState

### 저장 원칙

- 메모 데이터와 화면 상태를 같은 타이밍에 반복 저장하지 않는다.
- 입력 중에는 draft만 갱신하고 저장은 debounce한다.
- 서버 동기화 큐와 메신저 상태는 메모 본문 저장과 분리한다.
- 설치형 프로그램은 PC 로컬 저장소를 원본으로 유지한다.
- Web은 서버 공유 문서를 원본으로 유지한다.

## 메신저 2.3 설계

메신저 2.3은 기존 2.2 전체 그룹 메신저를 유지하면서 새 구조로 확장한다.

### 기능 범위

필수:

- 전체 그룹 채팅방 유지
- 채팅방 목록
- 일부 그룹원 대상 채팅방
- 채팅방별 unread count
- 텍스트 메시지 전송
- 파일 첨부
- 사진 첨부
- 이미지 미리보기
- 파일 크기 제한
- 확장자/MIME 제한
- 첨부 다운로드

보류:

- 메시지 수정
- 메시지 삭제
- 첨부 만료
- 참여자 나가기
- 모바일 메신저
- 설치형 메신저

### DB 후보

```text
messenger_rooms
messenger_room_members
messenger_messages
messenger_attachments
```

기존 2.2 데이터는 즉시 삭제하지 않는다.

전환 방식:

1. 새 테이블 추가
2. 전체 그룹 채팅방 자동 생성
3. 기존 그룹 메시지는 마이그레이션 스크립트로 새 전체 그룹 채팅방에 연결
4. 마이그레이션 전후 메시지 수 검증
5. 실패 시 기존 2.2 조회 API로 되돌릴 수 있게 유지

### API 후보

| 메서드 | 경로 | 목적 |
| --- | --- | --- |
| `GET` | `/api/v1/messenger/rooms` | 내 채팅방과 unread 조회 |
| `POST` | `/api/v1/messenger/rooms` | 일부 그룹원 채팅방 생성 |
| `GET` | `/api/v1/messenger/rooms/{room_id}/messages` | 메시지 조회 |
| `POST` | `/api/v1/messenger/rooms/{room_id}/messages` | 메시지 전송 |
| `POST` | `/api/v1/messenger/rooms/{room_id}/attachments` | 첨부 업로드 |
| `GET` | `/api/v1/messenger/attachments/{attachment_id}` | 첨부 다운로드 |
| `POST` | `/api/v1/messenger/rooms/{room_id}/read` | 읽음 처리 |
| `GET` | `/api/v1/messenger/policy` | 첨부 제한 정책 조회 |

### 첨부 정책

초기 기본값:

- 최대 파일 크기: 10MB
- 이미지: `jpg`, `jpeg`, `png`, `webp`, `gif`
- 일반 파일: `pdf`, `txt`, `md`, `docx`, `xlsx`, `pptx`, `zip`
- 차단: 실행 파일, 스크립트, 경로 포함 파일명, MIME 불일치
- 저장 경로: 서버 저장소의 messenger 전용 디렉터리
- 원본 파일명은 표시용 메타데이터로만 보관
- 실제 저장 파일명은 서버가 생성한 안전한 키를 사용

## 운영 화면/API 기준

2.3은 운영형 시스템 기준으로 설계한다.

운영자가 확인해야 하는 항목:

- 서버 저장소 사용량
- 메신저 첨부 총 용량
- 사용자별 업로드 용량
- 채팅방 수
- 메시지 수
- 첨부 파일 수
- DB에는 있으나 실제 파일이 없는 항목
- 실제 파일은 있으나 DB 메타데이터가 없는 항목
- 최근 동기화 실패
- 클라이언트별 마지막 접속 시각

확인 방식:

- 관리자 화면
- 관리자 API
- 화면 경고

금지:

- 운영자가 DB에 직접 접속해서 확인
- 운영자가 Python 명령어를 직접 실행해야만 확인 가능

## 성능 기준

2.3 완료 기준은 기능 동작뿐 아니라 성능 기준을 포함한다.

| 항목 | 기준 |
| --- | --- |
| idle CPU | Web을 켜두고 아무 작업이 없을 때 반복 작업이 없어야 함 |
| 입력 지연 | 제목/본문/검색/본문찾기 입력 중 포커스 이탈 없음 |
| 메신저 닫힘 | 메시지 목록 polling 없음 |
| unread count | 가벼운 API로만 조회 |
| 렌더링 | 입력 중 전체 render 금지 |
| 작은 창 | 제목과 본문 편집 가능 영역 유지 |
| 1,000개 메모 | 검색/선택/입력 사용 가능 |
| 1,000개 메시지 | 열린 방 최근 메시지만 렌더링 |

## 완료 기준

2.3은 아래 조건을 모두 만족해야 완료로 본다.

- 제목, 본문, 검색, 본문찾기 입력이 Web과 설치형에서 모두 안정적이다.
- 단축키가 문서화된 기준대로 동작한다.
- 작은 창에서도 편집이 깨지지 않는다.
- Web idle 상태에서 시스템 전체가 느려지지 않는다.
- 메신저가 닫혀 있으면 메시지 목록을 계속 가져오지 않는다.
- unread count가 표시된다.
- 전체 그룹 메신저가 유지된다.
- 일부 그룹원 채팅방을 만들 수 있다.
- 파일/사진 첨부가 제한 정책과 권한 검사를 통과한다.
- 설치형 프로그램은 별도 exe로 빌드된다.
- Web 전용 기능은 설치형에서 보이지 않는다.
- 운영 화면/API로 저장소와 이상 상태를 확인할 수 있다.

## 우선 결정 사항

개발 착수 전 확정할 항목:

- `Ctrl+F`: 통합 검색으로 확정
- 본문찾기 기본 단축키: `Ctrl+Shift+F`로 확정
- 첨부 기본 크기 제한: 10MB 후보
- 설치형 메신저: 2.3 제외
- 모바일 메신저: 2.3 제외
- 메신저 첨부 저장소: 메모 첨부와 분리
- 기존 2.2 메시지: 보존 후 마이그레이션

## 다음 작업
