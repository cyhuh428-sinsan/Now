# NowNote 2.3 공통 기준표와 릴리즈 검증표

이 문서는 NowNote 2.3부터 서버, Web, 설치형 프로그램, 모바일 앱을 나누어 작업하되 같은 제품 기준으로 릴리즈하기 위한 운영 기준표다.

## 기본 원칙

- 저장소와 제품 버전은 하나로 관리한다.
- 작업 단위는 서버, Web, 설치형 프로그램, 모바일 앱으로 분리한다.
- 기능 기준, API 계약, 릴리즈 판정은 하나의 기준표로 관리한다.
- 어느 한 프로그램만 수정됐다고 전체 버전 완료로 보지 않는다.
- exe와 apk는 릴리즈마다 GitHub Release asset으로 별도 업로드한다.
- 배포 전에는 실제 사용 흐름 기준으로 검증한다.

## UI 기준

| 항목 | 기준 |
| --- | --- |
| 최상위 제목 | 화면 또는 패널의 최상위 제목이며 14px 기준 |
| 제목 | 카드, 섹션, 목록 항목의 제목이며 12px 기준 |
| 본문 | 메모 본문과 일반 목록 내용이며 11px 또는 10px 기준 |
| 설명 | 보조 설명, 상태, 안내 문구이며 9px 기준 |
| 아이콘 메뉴 | 좁은 화면에서 버튼 텍스트가 영역을 차지하면 아이콘 또는 메뉴로 접는다. |
| 작은 창 | 설치형 프로그램은 작은 창에서도 주요 입력과 본문 영역이 깨지지 않아야 한다. |
| Web 전용 기능 | 설치형과 모바일에 그대로 넣지 않는다. 단, 디자인 흐름은 최대한 맞춘다. |

## 성능 기준

| 항목 | 기준 |
| --- | --- |
| Web 초기 실행 | 메모 앱답게 가볍게 떠야 하며 시스템 전체를 느리게 만들면 실패 |
| 메신저 갱신 | 짧은 polling 남발 금지, 중복 타이머 금지, 화면이 닫히면 중지 |
| 입력 반응성 | 제목, 본문, 검색, 본문찾기 입력이 끊기거나 막히면 실패 |
| 백그라운드 작업 | 저장, 동기화, 메신저 갱신은 중앙 스케줄러 기준으로 통제 |
| 긴 목록 | 필요 시 지연 렌더링 또는 표시 범위 제한을 적용 |

## 작업 분리 규칙

- 서버 작업자는 클라이언트 UI를 임의로 바꾸지 않는다.
- Web/설치형 작업자는 API 응답 형식을 임의로 바꾸지 않는다.
- 앱 작업자는 Web 수정이 자동 반영된다고 가정하지 않는다.
- 전체 기준 담당자는 각 영역 완료 표시 전에 산출물과 검증 결과를 확인한다.
- 한 영역에서 실패가 있으면 전체 버전 완료로 표시하지 않는다.

## 2.3 현재 위험 항목

- Web과 설치형은 코드가 비슷하지만 완전히 같은 파일을 쓰지 않는다.
- 모바일 앱은 Flutter 별도 구현이므로 Web/설치형 수정이 자동 반영되지 않는다.
- exe 파일명과 제품 버전 표시가 불일치할 수 있다.
- GitHub Release asset 업로드 여부와 로컬 빌드 생성 여부는 별도로 확인해야 한다.
- 실제 브라우저 입력 검증 없이 정적 검증만 통과하면 입력 문제를 놓칠 수 있다.

## 담당자 운영 기준

- 어울은 전체 기준표, 버전, 릴리즈 검증 담당으로 둔다.
- 각 작업자는 수정 전 담당 영역과 제외 영역을 명시한다.
- 수정 후에는 담당 영역 검증 결과를 기준표 항목에 맞게 보고한다.
- 릴리즈 전 최종 판정은 이 문서의 릴리즈 완료 조건을 기준으로 한다.
