# NowNote 2.3 인증/서버 연결 정리 작업지시서

## 목적

NowNote의 서버, Web, 설치형 프로그램, 모바일 앱 인증 구조가 많아져 사용자가 혼동하고 있다.

2.3에서는 내부 인증 구조를 무리하게 하나로 합치지 않는다. 대신 사용자 화면과 문서에서 역할을 명확히 나누고, Web/설치형/앱의 입력 항목과 설명을 같은 기준으로 정리한다.

## 확정 기준

| 대상 | 접속 방식 | 사용 토큰 | 설명 |
| --- | --- | --- | --- |
| Web | 사용자 ID / 비밀번호 로그인 | `X-Now-Web-Session` | 공용 서버 Web 화면에서 서버 공유 문서를 사용한다. |
| 설치형 프로그램 | 서버 API 연결 | `X-Now-User-Token` | 로컬 메모를 우선 사용하고, 필요 시 서버와 동기화한다. |
| 모바일 앱 | 서버 API 연결 | `X-Now-User-Token` | 모바일 메모, 녹음, 분석 작업을 서버와 동기화한다. |
| 운영자/구형 개인 서버 | 구형/특수 API 보호 | `NOW_API_TOKEN` | 2.3 기본 사용자 흐름에서는 제거한다. 필요한 경우에만 고급 호환 설정으로 둔다. |

## 사용자에게 보여줄 기본 흐름

### Web 사용자

1. `https://nownote.sinsan.kr` 접속
2. 사용자 ID와 비밀번호로 로그인
3. 서버 공유 문서, 그룹, 메신저 등 Web 전용 기능 사용
4. 앱/설치형 접속 토큰이 필요하면 Web에서 발급

### 설치형 프로그램 사용자

1. 설치형 프로그램 실행
2. 화면 설정에서 서버 연결 선택
3. 서버 주소 입력
4. 사용자 ID 입력
5. 앱/설치형 접속 토큰 입력
6. 2단계 인증이 필요한 계정은 연결 테스트 때만 6자리 코드 입력
7. 자동 동기화 필요 시 켬

### 모바일 앱 사용자

1. 앱 실행
2. 서버 설정 화면에서 서버 주소 입력
3. 사용자 ID 입력
4. 앱/설치형 접속 토큰 입력
5. 2단계 인증이 필요한 계정은 연결 테스트 때만 6자리 코드 입력
6. 동기화 실행

## 화면 표시 기준

### 기본 표시 항목

설치형 프로그램과 모바일 앱의 서버 설정 기본 화면에는 아래 항목을 우선 표시한다.

- 사용 방식
- 서버 주소
- 앱/설치형 접속 토큰
- 2단계 인증 코드
- 사용자 ID
- 자동 동기화
- 사용자 프로필
- 연결 테스트/저장/동기화 버튼

### 고급 표시 항목

아래 항목은 기본 사용자가 먼저 보지 않도록 접거나 고급 설정으로 분리한다.

- 개인 서버 API 토큰
- 기기 ID
- 운영자용 설명
- 개인 서버 관리자 보호 설명

개인 서버 API 토큰은 2.3 기본 흐름에서는 필요 없는 항목으로 본다. 다만 기존 개인 서버나 구형 배포 환경에서 `NOW_API_TOKEN` 보호를 이미 쓰고 있을 수 있으므로 즉시 삭제하지 않고 `구형 개인 서버 호환용` 고급 설정으로 숨긴다.

## 명칭 기준

| 기존 표현 | 2.3 기준 표현 | 이유 |
| --- | --- | --- |
| API 토큰 | 구형 개인 서버 API 토큰 | 2.3 기본 흐름에서는 필요 없으며, 기존 개인 서버 호환용임을 명확히 표시 |
| 사용자별 접속 토큰 | 앱/설치형 접속 토큰 | Web 로그인용이 아니라 앱/설치형 동기화용임을 명확히 표시 |
| 사용자 ID | 사용자 ID | 유지 |
| 기기 ID | 기기 ID | 고급 설정으로 이동 가능 |
| 2단계 인증 코드 | 2단계 인증 코드 | 저장하지 않고 연결 테스트 때만 사용 |

## 서버 기준

서버는 아래 API 구조를 유지한다.

| API | 대상 | 유지 여부 |
| --- | --- | --- |
| `/api/v1/auth/web-login` | Web ID/password 로그인 | 유지 |
| `/api/v1/auth/web-session` | Web 세션 확인 | 유지 |
| `/api/v1/auth/web-logout` | Web 로그아웃 | 유지 |
| `/api/v1/auth/token-login` | 앱/설치형 접속 토큰 확인 | 유지 |
| `/api/v1/auth/client-login` | ID/password로 기기 토큰 발급 | 서버 기능은 유지하되 2.3 기본 사용자 흐름에는 노출하지 않음 |
| `/api/v1/auth/device-token` | Web 세션 기반 기기 토큰 발급 | 유지 |

## 서버 및 API 작업 지시

### 1단계: API 역할 유지

- Web 로그인용 `/api/v1/auth/web-login`을 유지한다.
- Web 세션 확인용 `/api/v1/auth/web-session`을 유지한다.
- Web 로그아웃용 `/api/v1/auth/web-logout`을 유지한다.
- 앱/설치형 접속 토큰 확인용 `/api/v1/auth/token-login`을 유지한다.
- 데이터 API는 앱/설치형 요청에서 `X-Now-User-Token`을 계속 검증한다.
- `NOW_API_TOKEN` 기반 `Authorization: Bearer ...` 처리는 구형 개인 서버 호환용으로 유지하되, 2.3 기본 사용자 흐름으로 안내하지 않는다.

### 2단계: 조회 문제 확인 API 기준

- 앱/설치형 조회 실패를 확인할 수 있도록 `/api/v1/auth/token-login` 응답이 사용자 ID, 그룹, 활성 상태를 명확히 반환하는지 확인한다.
- `/api/v1/sync` 또는 `/api/v1/notes/sync` 요청에서 `owner_id`, `device_id`, `X-Now-User-Token` 검증 실패 사유가 구분되는지 확인한다.
- Web 세션 전용 공유 문서와 앱/설치형 토큰 동기화 문서 범위가 서버 응답에서 혼동되지 않도록 유지한다.
- 서버 로그 또는 운영 화면에서 앱/설치형 접속 토큰 마지막 사용 시각을 확인할 수 있어야 한다.

### 3단계: 서버 문서 정리

- `docs/SERVER_AUTH_POLICY.md`에서 `NOW_API_TOKEN`을 기본 연결 수단이 아니라 구형 개인 서버 호환/운영 보호용으로 설명한다.
- `server/README.md`에서 공용 서버 일반 사용자가 API 토큰을 입력하지 않는다는 점을 명확히 한다.
- 서버 capability 설명에 Web 로그인, 앱/설치형 접속 토큰, 구형 개인 서버 API 토큰의 역할을 분리한다.

## Web 및 설치형 프로그램 작업 지시

### 1단계: Web 기준 유지

- Web 로그인은 사용자 ID / 비밀번호 방식을 유지한다.
- Web 로그인 화면에는 앱/설치형 접속 토큰 입력을 노출하지 않는다.
- Web 설정 화면의 서버 연결 영역에서는 토큰 입력이 필요할 때만 보이도록 유지한다.
- 앱/설치형 접속 토큰 발급 화면 안내를 명확히 한다.
- 사용자에게 “이 토큰은 앱/설치형 서버 연결 설정에 붙여넣는 값”이라고 표시한다.

### 2단계: 설치형 기준 유지

- 현재 설치형 프로그램은 서버 API 연결 구조를 유지한다.
- ID/password 로그인 구조로 변경하지 않는다.
- 서버 설정 설명에서 Web 로그인과 설치형 연결 방식을 명확히 분리한다.
- 개인 서버 API 토큰은 기본 화면에서 제거하고 `구형 개인 서버 호환용` 고급 설정으로 숨긴다.
- 앱/설치형 접속 토큰은 동기화용 개인 토큰이라는 설명을 유지한다.
- 2단계 인증 코드는 연결 테스트 때만 입력한다는 설명을 유지한다.

### 3단계: Web/설치형 검증

- Web에서 ID/password 로그인 후 서버 공유 문서 조회가 유지되는지 확인한다.
- Web에서 앱/설치형 접속 토큰 발급 안내가 보이는지 확인한다.
- 설치형에서 서버 주소, 사용자 ID, 앱/설치형 접속 토큰으로 연결 테스트가 되는지 확인한다.
- 설치형에서 구형 개인 서버 API 토큰이 기본 화면이 아니라 고급 설정에만 보이는지 확인한다.
- 설치형 동기화 요청에 `X-Now-User-Token`이 포함되는지 확인한다.

## 앱 작업 지시

### 1단계: 현재 화면 기준 맞추기

- 모바일 앱 서버 설정 화면의 `사용자별 접속 토큰` 문구를 `앱/설치형 접속 토큰`으로 바꾼다.
- 모바일 앱 서버 설정 화면의 `API 토큰` 문구를 `구형 개인 서버 API 토큰`으로 바꾼다.
- 구형 개인 서버 API 토큰은 기본 입력 영역에서 제거하고 고급 설정 영역으로 숨긴다.
- 기기 ID도 가능하면 고급 설정 영역으로 분리한다.
- 설명 문구를 설치형 프로그램과 같은 기준으로 맞춘다.

### 2단계: 조회 문제 확인

앱에서 “조회가 잘 안 되는 부분”은 아래 순서로 원인을 좁힌다.

1. 서버 주소가 실제 앱에서 접근 가능한 주소인지 확인
2. 사용자 ID가 Web 계정의 `owner_id`와 같은지 확인
3. 앱/설치형 접속 토큰이 해당 사용자에게 발급된 토큰인지 확인
4. 연결 테스트가 `/api/v1/auth/token-login`까지 통과하는지 확인
5. 동기화 요청에 `X-Now-User-Token`이 포함되는지 확인
6. 일자별 메모와 계층 메모가 서버에 올라가는지 확인
7. 서버에서 내려오는 메모가 앱 로컬 DB에 반영되는지 확인
8. Web 세션 전용 공유 문서나 그룹 메모를 앱에서 기대하고 있지 않은지 확인

### 3단계: 앱 검증 보강

`now_app/scripts/verify_mobile_surface.py`에 아래 정적 검증을 추가한다.

- `앱/설치형 접속 토큰` 문구 존재
- `구형 개인 서버 API 토큰` 문구가 고급 설정에만 존재
- `/api/v1/auth/token-login` 호출 유지
- `X-Now-User-Token` 헤더 유지
- `Authorization: Bearer`는 기본 흐름이 아니라 구형 개인 서버 호환용으로만 유지
- 2단계 인증 코드는 저장 필드가 아니라 연결 테스트 전달용으로 유지

- `README.md`
- `docs/HELP.md`
- `docs/SERVER_AUTH_POLICY.md`
- `web/README.md`
- `desktop/README.md`

반드시 반영할 문장 기준:

- Web은 사용자 ID와 비밀번호로 로그인한다.
- 설치형 프로그램과 모바일 앱은 서버 주소, 사용자 ID, 앱/설치형 접속 토큰으로 서버에 연결한다.
- 개인 서버 API 토큰은 2.3 기본 사용 흐름에서는 필요 없으며, 기존 개인 서버 호환이 필요할 때만 고급 설정에서 사용한다.
- 2단계 인증 코드는 저장하지 않고 연결 확인 때만 입력한다.

## 완료 조건

아래 조건을 모두 만족해야 이 작업을 완료로 본다.

- 앱 서버 설정 화면 문구가 설치형 기준과 맞음
- 앱에서 구형 개인 서버 API 토큰이 기본 흐름에서 숨겨지고 앱/설치형 접속 토큰이 기본 연결 수단으로 표시됨
- 앱 조회 문제 원인이 토큰, 사용자 ID, 서버 주소, 동기화 범위 중 어디인지 확인됨
- Web은 ID/password 로그인 기준을 유지함
- 설치형은 API 연결 기준을 유지함
- 문서의 인증/연결 설명이 서로 충돌하지 않음
- 모바일 정적 검증 통과
- Flutter 테스트 통과
- Web 표면 검증 통과
- 설치형 저장/단축키/서버 연결 검증 통과

## 금지 사항

- 설치형 프로그램을 ID/password 로그인 방식으로 바꾸지 않는다.
- 모바일 앱을 ID/password 로그인 방식으로 바꾸지 않는다.
- 구형 개인 서버 API 토큰 기능을 확인 없이 즉시 삭제하지 않는다.
- 앱/설치형 접속 토큰 검증 API를 제거하지 않는다.
- Web 세션 전용 기능을 앱/설치형에 그대로 노출하지 않는다.
- 조회 문제를 확인하지 않은 상태에서 동기화 구조를 임의로 바꾸지 않는다.

1. 앱 서버 설정 문구와 입력 구조 정리

3. 앱 검증 스크립트 보강
