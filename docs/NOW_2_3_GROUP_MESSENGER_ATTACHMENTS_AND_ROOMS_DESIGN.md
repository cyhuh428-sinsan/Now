# NowNote 2.3 그룹 메신저 첨부와 채팅방 후보 설계서

## 문서 상태

이 문서는 2.3 개발을 바로 시작하기 위한 구현 명세가 아니라, 후보 기능을 모으고 설계 기준을 확정하기 위한 초안이다.

2.3 구현은 `NOW_2_3_LIGHTWEIGHT_RUNTIME_REDESIGN.md`의 경량 런타임 기준을 먼저 만족한 뒤 진행한다. 메신저 확장은 메모 입력, 제목 수정, 검색, 본문찾기의 반응성을 해치면 완료로 보지 않는다.

2.2의 전체 그룹 메신저, 자동 새로고침, 안 읽은 메시지 수, 읽음 처리는 유지한다. 2.3은 그 위에 파일/사진 첨부와 일부 그룹원 대상 채팅방을 추가할지 검토한다.

## 목적

2.3의 목표는 hosted Web 그룹 메신저를 단순 전체 그룹 대화에서 운영형 협업 대화 도구로 확장하는 것이다.

핵심 방향은 두 가지다.

- 메시지에 파일과 사진을 안전하게 첨부한다.
- 전체 그룹 대화와 별도로, 그룹원 일부만 참여하는 채팅방을 만든다.

## 진행 원칙

- 후보 기능을 먼저 모은 뒤 필수, 보류, 제외로 나눈다.
- 설계 확정 전에는 DB/API/Web 구현을 시작하지 않는다.
- 기존 2.2 전체 그룹 메신저는 제거하거나 의미를 바꾸지 않는다.
- 메모 본문 첨부와 메신저 첨부 저장소는 분리한다.
- 파일 접근 권한은 같은 그룹이 아니라 같은 채팅방 참여자를 기준으로 제한한다.
- 운영자가 DB/CLI 없이 화면과 API로 저장소 상태를 확인할 수 있어야 한다.

## 제외 후보

아래 항목은 2.3 확정 전에는 제외로 본다.

- 외부 push 알림
- 모바일 앱 메신저
- 설치형 프로그램 메신저
- 음성/영상 통화
- 공개 링크로 파일 공유
- 그룹 밖 사용자 초대
- 종단 간 암호화 첨부
- 대용량 파일 전송 서비스 수준의 기능

## 사용자 흐름

### 전체 그룹 메신저

1. 사용자가 hosted Web에 로그인한다.
2. `그룹 메신저`에서 전체 그룹 대화방을 선택한다.
3. 같은 그룹의 모든 활성 사용자가 메시지를 조회한다.
4. 사용자가 텍스트 또는 첨부가 있는 메시지를 보낸다.
5. 같은 그룹 사용자에게 새 메시지와 안 읽은 수가 반영된다.

### 일부 그룹원 채팅방

1. 사용자가 `새 채팅방`을 선택한다.
2. 같은 그룹의 활성 사용자 목록에서 참여자를 고른다.
3. 채팅방 이름을 입력하거나 참여자 기준 기본 이름을 사용한다.
4. 채팅방을 만든 사용자는 자동 참여자가 된다.
5. 채팅방 참여자만 메시지와 첨부를 조회한다.

### 파일/사진 첨부

1. 사용자가 메시지 입력 영역에서 파일 또는 사진을 선택한다.
2. Web은 업로드 전 크기와 확장자 1차 검사를 수행한다.
3. 서버는 Web 검사와 별도로 크기, MIME type, 확장자, 채팅방 참여 권한을 다시 검사한다.
4. 업로드 성공 후 메시지에 첨부 메타데이터를 연결한다.
5. 이미지 파일은 미리보기를 표시하고, 일반 파일은 파일명/크기/다운로드 버튼을 표시한다.

## 화면 설계 초안

### 메신저 패널

- 왼쪽 또는 상단에 채팅방 목록을 둔다.
- 첫 항목은 항상 `전체 그룹` 채팅방이다.
- 각 채팅방 항목에는 이름, 최근 메시지 시각, 안 읽은 수를 표시한다.
- 채팅방을 선택하면 오른쪽 메시지 목록과 입력창이 바뀐다.

### 새 채팅방 화면

- 같은 그룹의 활성 사용자 목록을 체크박스로 표시한다.
- 검색으로 사용자를 좁힐 수 있다.
- 채팅방 이름은 선택 입력으로 둔다.
- 최소 참여자는 본인 외 1명으로 한다.

### 메시지 입력 영역

- 텍스트 입력
- 파일 선택 버튼
- 사진 선택 버튼
- 선택된 첨부 목록
- 전송 버튼
- 업로드 실패 메시지

## 데이터 구조 후보

| `created_at` | 생성 시각 |
| `updated_at` | 최근 활동 시각 |
| `is_active` | 사용 여부 |

| 메서드 | 경로 | 목적 |
| --- | --- | --- |

| `GET` | `/api/v1/messenger/rooms/{room_id}/messages` | 채팅방 메시지 조회 |

## 저장소 정책 후보

| 항목 | 후보 기준 |
| --- | --- |
| 기본 저장소 | 서버 `NOW_STORAGE_DIR` 아래 messenger 전용 하위 경로 |
| 경로 규칙 | 사용자 입력 파일명을 경로로 사용하지 않음 |
| 파일명 표시 | 원본 파일명은 DB 메타데이터로만 보관 |
| 크기 제한 | 기본 10MB 후보, 운영 설정으로 조정 가능 |
| 이미지 확장자 | `jpg`, `jpeg`, `png`, `webp`, `gif` 후보 |
| 일반 파일 확장자 | `pdf`, `txt`, `md`, `docx`, `xlsx`, `pptx`, `zip` 후보 |
| 차단 | 실행 파일, 스크립트, 경로 포함 파일명, MIME 불일치 |
| 백업 | 첨부 메타데이터와 실제 파일 누락/고아 파일 점검 필요 |

## 권한 기준

- Web 로그인 세션이 없으면 모든 메신저 API를 거부한다.
- 채팅방 생성자는 반드시 자기 그룹의 활성 사용자여야 한다.
- 참여자로 추가할 수 있는 사용자는 같은 그룹의 활성 사용자만 허용한다.
- 메시지 조회, 전송, 첨부 업로드, 다운로드는 채팅방 참여자만 가능하다.
- 전체 그룹 채팅방은 같은 그룹의 활성 사용자 전체를 참여자로 본다.
- 관리자라도 일반 다운로드 API에서는 참여자 권한을 우회하지 않는다. 운영 점검은 별도 관리자 API로 분리한다.

## 운영 화면 후보

- 전체 메신저 저장소 사용량
- 사용자별 업로드 용량
- 채팅방 수와 메시지 수
- 첨부 파일 수와 총 용량
- 고아 첨부 파일 점검
- DB 메타데이터는 있으나 실제 파일이 없는 항목 점검
- 파일 제한 정책 표시

## 2.2에서 2.3으로 넘어가는 기준

- 기존 `group_messages`와 `group_message_reads`를 그대로 유지할지, 새 `messenger_rooms` 구조로 마이그레이션할지 결정한다.
- 전체 그룹 메신저는 새 구조에서도 기본 채팅방으로 보여야 한다.
- 2.2 사용자는 업그레이드 후 기존 메시지를 잃지 않아야 한다.
- 마이그레이션이 필요하면 서버 시작 시 자동 보정이 아니라 명시적 마이그레이션 파일로 처리한다.

## 확정 전 질문

- 파일 크기 기본 제한을 10MB로 둘지, 더 작게 시작할지 결정해야 한다.
- 허용 확장자 목록을 운영자가 설정할 수 있게 할지 결정해야 한다.
- 채팅방 이름을 필수로 할지, 참여자 이름 기반 자동 이름을 기본으로 할지 결정해야 한다.
- 참여자 추가/나가기를 2.3에 포함할지 보류할지 결정해야 한다.
- 메시지 삭제와 첨부 삭제를 2.3에 포함할지 보류할지 결정해야 한다.
- 이미지 썸네일을 서버가 생성할지, 원본을 브라우저에서 축소 표시할지 결정해야 한다.

## 완료 기준 후보

2.3 개발을 시작하기 전 완료 기준은 아래처럼 확정한다.

- 후보 기능이 필수, 보류, 제외로 분류되어 있다.
- DB 마이그레이션 방향이 확정되어 있다.
- API 목록과 권한 기준이 확정되어 있다.
- Web 화면 흐름이 확정되어 있다.
- 저장소 경로와 파일 제한 정책이 확정되어 있다.
- 기존 2.2 전체 그룹 메신저 유지 방식이 확정되어 있다.
- 운영 화면/API 점검 항목이 확정되어 있다.
