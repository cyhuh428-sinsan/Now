# NowNote 2.3 경량 런타임 재설계서

## 문서 상태

이 문서는 2.3 메신저 확장 작업 전에 반드시 확정해야 하는 경량화 기준이다.

2.3은 파일, 사진, 채팅방 기능을 추가하기 전에 Web과 설치형 프로그램이 메모 앱답게 가볍게 동작하도록 런타임 구조를 먼저 정리한다.

## 문제 정의

NowNote는 메모 프로그램이므로 항상 켜 두어도 다른 프로그램에 영향을 주면 안 된다.

현재 문제는 특정 기능 하나가 아니라, 메신저, 공유 조회, 서버 동기화 같은 부가 기능이 메모 편집 흐름과 같은 화면 상태에 붙어 전체 렌더링과 저장을 자극할 수 있다는 점이다.

## 설계 원칙

- 메모 입력과 제목 수정은 최우선이며 서버 작업 때문에 지연되면 안 된다.
- 보이지 않는 기능은 기본적으로 동작하지 않는다.
- 메신저는 메모 데이터 저장 루프와 분리한다.
- 서버 갱신은 변경분만 처리하고 같은 데이터는 다시 렌더링하지 않는다.
- 전체 트리, 전체 메시지, 전체 공유 목록을 반복 렌더링하지 않는다.
- 작은 창에서는 본문과 제목 영역을 우선 확보하고, 조작 버튼은 아이콘 또는 메뉴로 접는다.
- Web이 idle 상태일 때 CPU 사용량은 거의 없어야 한다.

## 2.3 작업 순서

1. 경량 런타임 기준 확정
2. 메모 코어와 부가 기능 상태 분리
3. 검색, 본문찾기, 제목 수정 입력 안정화
4. 메신저를 열린 채팅방 중심 구조로 분리
5. unread count는 가벼운 별도 API 또는 증분 응답으로 처리
6. 파일/사진 첨부와 일부 그룹원 채팅방 설계 확정
7. 2.3 메신저 확장 구현

## 메모 코어 기준

메모 코어는 아래 기능만 포함한다.

- 트리 선택
- 제목 수정
- 본문 수정
- 탭 열기와 닫기
- 로컬 저장
- 검색과 본문찾기 입력

메모 코어는 메신저 메시지 수신, 공유 문서 조회, 서버 상태 확인 때문에 다시 렌더링되면 안 된다.

## 메신저 기준

- 메신저 창이 닫혀 있으면 메시지 목록을 가져오지 않는다.
- 닫힌 상태에서는 unread count만 아주 가볍게 확인한다.
- 열린 채팅방만 메시지를 가져온다.
- 메시지는 최근 N개만 렌더링하고 과거 메시지는 추가 로딩으로 처리한다.
- 메시지 상태는 메모 데이터와 별도 저장소 또는 별도 메모리 상태로 둔다.
- 같은 응답이면 DOM 갱신과 저장을 하지 않는다.

## 서버 통신 기준

- 기능별 독립 polling을 금지한다.
- 하나의 scheduler가 화면 상태, 탭 활성 상태, 마지막 요청 시각을 기준으로 작업을 조절한다.
- 브라우저 탭이 비활성 상태면 갱신 주기를 늦춘다.
- 실패 재시도는 backoff를 적용한다.
- 사용자 입력 중에는 무거운 렌더링 작업을 지연한다.

## 화면 기준

- 작은 창에서도 제목 입력, 본문 입력, 검색 입력은 즉시 가능해야 한다.
- 상단 조작은 기본적으로 메뉴에 넣는다.
- 사용자가 설정하면 자주 쓰는 조작만 아이콘으로 표시한다.
- 긴 텍스트 버튼은 작은 창의 상단 편집 영역에 직접 배치하지 않는다.

## 성능 완료 기준

- Web 실행 후 아무 작업이 없으면 반복 렌더링이 없어야 한다.
- 메신저를 열지 않으면 메시지 DOM 렌더링이 없어야 한다.
- 제목 입력과 본문 입력 중 포커스가 다른 기능으로 이동하지 않아야 한다.
- 검색과 본문찾기는 열면 바로 입력 가능해야 한다.
- 1,000개 메모와 1,000개 메시지 조건에서도 메모 입력 지연이 없어야 한다.

## 2.3 메신저 확장과의 관계

`NOW_2_3_GROUP_MESSENGER_ATTACHMENTS_AND_ROOMS_DESIGN.md`의 파일, 사진, 채팅방 기능은 이 문서의 경량 런타임 기준을 만족한 뒤 구현한다.

2.3에서 기능을 추가하더라도 메모 코어를 무겁게 만들면 완료로 보지 않는다.
