# NowNote 서버 복구 절차

이 문서는 개인 Docker 서버와 공용 NowNote 서버에서 장애가 발생했을 때 운영자가 확인할 순서를 정리합니다.

현재 1차 서버는 백업 JSON 내보내기와 백업 검증 API를 제공합니다. 자동 복원 API는 아직 제공하지 않습니다. 복원은 운영자가 백업 JSON을 검증한 뒤 DB와 저장소 상태를 확인하고 수동 절차로 진행하는 것을 기준으로 합니다.

## 복구 전 확인

1. 현재 서버를 중지하기 전에 가능하면 `/admin/export`에서 전체 백업을 내려받습니다.
2. 내려받은 백업 JSON을 `POST /api/v1/admin/export/verify`로 검증합니다.
3. 검증 결과가 `ok`인지 확인합니다.
4. `status_counts.bad`가 0인지 확인합니다.
5. `checks`에 `bad`가 있으면 복구 작업을 시작하지 말고 백업 파일을 다시 생성하거나 다른 백업본을 확인합니다.
6. `checks`에 `warn`이 있으면 `/admin/ops`에서 DB, 저장소, 토큰, 분석 작업 상태를 먼저 확인합니다.
7. `summary`의 메모, 녹음 메타데이터, 사용자, 분석 작업, 동기화 이력 건수를 기록합니다.
8. `summary.recording_orphan_files`와 `summary.recording_orphan_bytes`가 0보다 크면 `/api/v1/admin/export/recording-orphans` 또는 `/admin/recordings`의 고아 녹음 파일 JSON을 별도로 저장합니다.
9. `summary.recording_missing_files`가 0보다 크거나 `/admin/ops`의 `누락 녹음 파일` 항목이 `bad`이면 `/api/v1/admin/export/recording-missing-files`를 내려받고, 저장소 백업에서 해당 원본 파일을 먼저 찾습니다.

## 백업 파일 기준

전체 백업 JSON에는 다음이 포함됩니다.

- 메모
- 녹음 메타데이터
- 사용자 계정 운영 메타데이터
- 분석 작업
- 동기화 이력
- 삭제 표시 메모

전체 백업 JSON에는 다음이 포함되지 않습니다.

- 원본 녹음 파일
- 사용자별 접속 토큰 원문
- 사용자별 접속 토큰 해시
- Web에서 발급한 앱/설치형 연결 토큰 원문

원본 녹음 파일은 Docker 볼륨 또는 `NOW_STORAGE_DIR`에 매핑된 저장소를 별도로 보존해야 합니다.
고아 녹음 파일은 전체 백업 JSON에 포함되지 않는 저장소 잔여 파일입니다. 복구 전에는 자동 삭제하지 말고 목록을 별도 보관한 뒤, DB 메타데이터와 연결할지 수동 정리할지 판단합니다.
누락 녹음 파일은 백업 JSON에는 녹음 메타데이터가 있지만 저장소 원본 파일이 없는 상태입니다. 복구 전에는 DB 메타데이터를 삭제하지 말고 저장소 백업, Docker 볼륨 백업, 이전 서버의 `NOW_STORAGE_DIR`에서 원본 파일을 먼저 찾습니다.

## Docker 서버 점검 순서

1. `.env`의 `NOW_API_TOKEN`, `NOW_POSTGRES_PASSWORD`, `NOW_STORAGE_DIR` 값을 확인합니다.
2. `docker-compose.yml`에서 Postgres 볼륨과 녹음 저장소 볼륨이 의도한 경로인지 확인합니다.
3. 컨테이너를 재시작하기 전 DB 볼륨과 녹음 저장소를 별도로 복사합니다.
4. `docker compose` 또는 `docker-compose`로 API, worker, Postgres 컨테이너 상태를 확인합니다.
5. 서버가 올라오면 `/health`, `/health/ready`, `/api/v1/server`를 확인합니다.
6. `/admin/ops`에서 DB, 저장소, 토큰, 분석 작업 상태를 확인합니다.
7. `scripts/smoke_test.py`를 실행해 동기화, 내보내기, 백업 검증이 정상인지 확인합니다.

## 복원 판단 기준

다음 조건을 모두 만족할 때만 백업 기반 복원을 진행합니다.

- 백업 검증 API가 `ok`를 반환합니다.
- 백업 검증 결과의 `checks`에 `bad`가 없습니다.
- `warn` 항목이 있으면 `/admin/ops`에서 원인을 확인했고 복구에 영향이 없다고 판단했습니다.
- 백업 파일의 `content_sha256`이 응답 헤더 또는 기록된 값과 일치합니다.
- 백업 파일의 `backup_schema_version`이 현재 서버에서 지원하는 값입니다.
- 원본 녹음 파일이 필요한 경우 저장소 볼륨 백업도 함께 확보되어 있습니다.
- 고아 녹음 파일이 있는 경우 목록 JSON과 저장소 원본을 함께 확보했습니다.
- 누락 녹음 파일이 있는 경우 누락 목록 JSON을 확보했고, 저장소 백업에서 복구 가능 여부를 확인했습니다.
- 복원 대상 서버의 DB와 저장소를 먼저 별도 백업했습니다.

## 공용 서버 주의사항

공용 서버에서는 복원 전에 사용자별 영향 범위를 먼저 확인해야 합니다.

- 특정 사용자만 영향을 받는지 확인합니다.
- 전체 DB 복원이 필요한지, 사용자 단위 조치가 가능한지 구분합니다.
- 백업 JSON에는 사용자별 접속 토큰과 Web 발급 앱/설치형 연결 토큰이 포함되지 않으므로 복원 후 필요하면 Web 또는 관리자 화면에서 새로 발급합니다.
- Web 직접 가입, 이메일 비밀번호 재설정, 2단계 인증, reverse proxy, 사용자별 데이터 격리 점검이 모두 통과해야 공용 서버 정식 오픈 대상으로 봅니다.
